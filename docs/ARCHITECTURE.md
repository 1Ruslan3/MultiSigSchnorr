# Архитектура проекта MultiSigSchnorr

## Общая характеристика архитектуры

Проект `MultiSigSchnorr` построен по многослойной архитектуре. Такое разделение позволяет изолировать криптографическую логику, доменную модель, прикладные сценарии, инфраструктуру хранения, API и пользовательский интерфейс.

Основная идея архитектуры заключается в том, что каждый слой отвечает только за свою область:

```text
Domain          -> предметная область
Crypto          -> криптографические операции
Protocol        -> протокольная логика коллективной подписи
Application     -> сценарии использования
Infrastructure  -> хранение и внешние зависимости
Contracts       -> DTO-модели обмена
Api             -> HTTP API
Web             -> пользовательский интерфейс
Tests           -> автоматическая проверка
Benchmarks      -> измерение производительности
```

Такое разделение упрощает сопровождение проекта и позволяет использовать его как демонстрационную и исследовательскую систему в рамках выпускной квалификационной работы.

## `MultiSigSchnorr.Domain`

### Назначение

Слой `Domain` содержит основные сущности и типы предметной области. Он не должен зависеть от API, Web, EF Core или других инфраструктурных механизмов.

### Основные элементы

В доменном слое находятся:

- участники протокола;
- эпохи;
- членство участников в эпохах;
- записи commitment;
- записи nonce reveal;
- частичные подписи;
- агрегированная подпись;
- аудит;
- статусы участников;
- статусы сессий;
- режимы защиты;
- value object-типы для скаляров, точек и ключей.

### Принцип

Доменный слой описывает, что существует в системе и какие инварианты должны соблюдаться.

Например:

- участник не должен иметь пустой идентификатор;
- эпоха должна иметь номер;
- commitment должен относиться к конкретной сессии и участнику;
- частичная подпись должна быть связана с участником;
- агрегированная подпись должна содержать aggregate nonce point и scalar.

---

## `MultiSigSchnorr.Crypto`

### Назначение

Слой `Crypto` содержит криптографические операции, которые используются протоколом коллективной подписи.

### Основные компоненты

В слое реализованы:

- контекст эллиптической кривой P-256;
- генерация открытого ключа;
- агрегирование открытых ключей;
- вычисление challenge;
- хеширование;
- генерация nonce;
- commitment service;
- вычисление частичной подписи;
- проверка агрегированной подписи;
- режимы обработки секретного скаляра.

### Режимы обработки

Поддерживаются два режима:

```text
Baseline
RandomizedScalarProcessing
```

`Baseline` используется как базовый режим.

`RandomizedScalarProcessing` используется для демонстрации защитной модификации, связанной с рандомизированной обработкой секретного скаляра.

### Роль в проекте

Криптографический слой не должен знать о Web, API, базе данных или пользовательских сценариях. Он предоставляет набор сервисов, которые вызываются протокольным слоем.

---

## `MultiSigSchnorr.Protocol`

### Назначение

Слой `Protocol` реализует логику коллективной цифровой подписи.

Он связывает криптографические операции в последовательный протокол:

```text
создание сессии
commitment
reveal nonce
partial signatures
aggregate signature
verification
```

### Основные задачи

Слой отвечает за:

- создание N-party протокольной сессии;
- проверку состава участников;
- вычисление агрегированного открытого ключа;
- генерацию nonce для участников;
- публикацию commitment-значений;
- раскрытие публичных nonce;
- вычисление challenge;
- формирование частичных подписей;
- агрегирование частичных подписей;
- проверку итоговой подписи;
- контроль переходов между состояниями.

### Протокольный цикл

Общий цикл выглядит так:

```text
1. Create Session
2. Publish Commitments
3. Reveal Nonces
4. Submit Partial Signatures
5. Build Aggregate Signature
6. Verify Aggregate Signature
```

### Связь с эпохами

Протокол использует механизм эпох. Эпоха определяет актуальный набор участников, которые могут участвовать в коллективной подписи.

Это позволяет:

- управлять составом участников;
- отзывать участников;
- переходить к новой эпохе;
- фиксировать изменения состава подписантов.

---

## `MultiSigSchnorr.Application`

### Назначение

Слой `Application` содержит сценарии использования системы. Он связывает доменную, криптографическую, протокольную и инфраструктурную части проекта.

### Основные use cases

В слое реализованы сценарии:

- создание протокольной сессии;
- публикация commitment;
- раскрытие nonce;
- отправка частичной подписи;
- получение состояния сессии;
- получение истории сессий;
- экспорт отчёта;
- проверка подписи;
- получение состояния администрирования;
- отзыв участника;
- переход к следующей эпохе;
- получение аудита.

### Репозитории

Слой `Application` определяет интерфейсы репозиториев:

- `IEpochRepository`;
- `IParticipantRepository`;
- `IEpochMemberRepository`;
- `IProtocolSessionRepository`;
- `IProtocolSessionProjectionRepository`;
- `IAuditLogRepository`;
- `IPrivateKeyMaterialRepository`.

Реализации этих интерфейсов находятся в `Infrastructure`.

### Принцип

`Application` не должен зависеть от конкретной базы данных. Он работает с абстракциями. Поэтому можно использовать in-memory-хранилище, PostgreSQL или другую реализацию.

---

## `MultiSigSchnorr.Infrastructure`

### Назначение

Слой `Infrastructure` содержит технические реализации:

- in-memory репозитории;
- PostgreSQL-репозитории;
- EF Core DbContext;
- persistence entities;
- миграции;
- доступ к базе данных.

### PostgreSQL

Через PostgreSQL сохраняются:

- аудит;
- эпохи;
- участники;
- членство в эпохах;
- публичная проекция протокольных сессий.

### In-memory

В памяти процесса остаются:

- runtime-состояние активной протокольной сессии;
- приватные ключевые материалы;
- secret nonce.

Это сделано намеренно, чтобы не сохранять секретные криптографические значения в базе данных.

---

## `MultiSigSchnorr.Contracts`

### Назначение

Слой `Contracts` содержит DTO-модели для обмена данными между API и Web.

### Примеры моделей

- request-модели создания протокольной сессии;
- response-модели состояния сессии;
- response-модели отчёта;
- response-модели аудита;
- response-модели диагностики;
- response-модели администрирования.

### Причина выделения слоя

Выделение контрактов позволяет:

- не отдавать наружу доменные сущности напрямую;
- стабилизировать API;
- отделить внутреннюю модель от внешнего представления;
- использовать одни и те же модели в API и Web.

---

## `MultiSigSchnorr.Api`

### Назначение

Слой `Api` предоставляет HTTP-интерфейс для работы с системой.

### Основные группы endpoints

```text
/api/system
/api/system/storage
/api/protocol-sessions
/api/admin
/api/audit
```

### Основные возможности API

API позволяет:

- получить seed-данные;
- получить диагностику PostgreSQL-хранилища;
- создать протокольную сессию;
- выполнить этапы протокола;
- получить состояние сессии;
- получить историю;
- получить отчёт;
- скачать отчёт в JSON/TXT;
- управлять эпохами и участниками;
- просматривать журнал аудита.

## `MultiSigSchnorr.Web`

### Назначение

Слой `Web` предоставляет пользовательский интерфейс для демонстрации системы.

### Основные страницы

```text
/system-overview
/protocol-sessions
/protocol-session-history
/administration
/audit-log
```

### Назначение страниц

#### `System Overview`

Показывает:

- seed-данные;
- активную эпоху;
- участников;
- PostgreSQL diagnostics;
- количество записей в таблицах.

#### `Protocol Sessions`

Позволяет:

- создать протокольную сессию;
- выбрать режим защиты;
- пройти этапы протокола;
- проверить подпись;
- сформировать отчёт;
- скачать отчёт.

Показывает историю протокольных сессий.

#### `Administration`

Позволяет управлять эпохами и участниками.

Показывает журнал аудита с фильтрацией и поиском.

## `Benchmarks`

### Назначение

Benchmark-проект используется для измерения производительности.

### Измеряются

- генерация открытого ключа в режиме `Baseline`;
- генерация открытого ключа в режиме `RandomizedScalarProcessing`;
- частичная подпись в обоих режимах;
- полный протокольный цикл в обоих режимах.

### Роль в ВКР

Бенчмарки позволяют оценить вычислительные накладные расходы защитного режима и использовать эти данные в практической части выпускной квалификационной работы.

## Persistence-подход

В проекте используется смешанная модель хранения:

```text
Публичное и отчётное состояние -> PostgreSQL
Секретное runtime-состояние    -> InMemory
```

### PostgreSQL хранит

- epochs;
- participants;
- epoch members;
- audit log entries;
- protocol session public projection;
- protocol session participants projection.

### PostgreSQL не хранит

- private key material;
- secret nonce;
- внутреннее runtime-состояние незавершённой сессии.

### Причина

Секретные криптографические значения не должны сохраняться в открытом виде в базе данных. Поэтому PostgreSQL используется для публичного состояния, отчётов и истории, а не для хранения секретов.

---

## Проекция протокольной сессии

Для сохранения протокольных сессий используется отдельная projection-модель.

Она хранит:

- session id;
- epoch id;
- статус;
- режим защиты;
- digest сообщения;
- aggregate public key;
- aggregate nonce point;
- challenge;
- aggregate signature;
- состояние участников;
- commitments;
- public nonce;
- partial signatures.

Эта модель достаточна для:

- просмотра истории;
- формирования отчётов;
- демонстрации публичных артефактов;
- анализа выполненных сессий.

Но она не предназначена для продолжения незавершённой сессии после перезапуска API.

---

## Итоговая архитектурная идея

Проект разделяет три уровня состояния:

```text
1. Доменная модель
   Описывает правила и сущности.

2. Runtime-состояние
   Используется во время выполнения протокола.

3. PostgreSQL-проекция
   Используется для истории, отчётов и аудита.
```

Такой подход позволяет сохранить безопасность секретных данных и при этом обеспечить воспроизводимую демонстрацию, аудит и сохранение публичного состояния.
